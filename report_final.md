# Stub-Coding TDD vs SDD — Claude 토큰 비용 및 코드 품질 비교

실험일: 2026-06-15
모델: Claude Sonnet 4.6 (SDD/TDD 스텁+테스트) / pi Qwen3.5 (TDD 구현)

---

## 1. 실험 설계 요약

| 항목 | 내용 |
|------|------|
| **SDD** | 스펙 → Claude가 테스트 + 구현 전부 작성 |
| **TDD** | 스펙 → Claude가 스텁 + 테스트만 작성 → pi(Qwen3.5)가 구현 |
| 측정 기준 | Claude Code `/cost` (세션 단위, 캐시 할인 반영된 실제 청구액) |
| 완료 기준 | pytest 전체 통과 |
| 시나리오 A | CSV 파서 + 통계 요약기 (iris_missing.csv) |
| 시나리오 B | DNA 서열 분석기 (tp53.fasta) |

## 2. 토큰 비용 비교

### 전체 수치

| 세션  | input | output | cache read | cache write | **총 비용** |
|-------|-------|--------|------------|-------------|------------|
| SDD-A | 353   | ~2.5k  | 183,200    | 26,300      | $0.2512    |
| TDD-A | 53    | ~2.5k  | 124,200    | 12,700      | **$0.1512** |
| SDD-B | 353   | ~2.9k  | 186,600    | 28,700      | $0.2740    |
| TDD-B | 351   | ~3.3k  | 134,700    | 26,100      | **$0.2479** |

### 비용 절감 효과

| 시나리오 | SDD | TDD | 절감액 | 절감률 |
|----------|-----|-----|--------|--------|
| A (CSV)  | $0.2512 | $0.1512 | $0.10 | **40%** |
| B (DNA)  | $0.2740 | $0.2479 | $0.026 | **9.5%** |

### 핵심 관찰: 왜 B에서 절감이 적었나?

```
Scenario A:  SDD input=353 / TDD input=53   → 85% 차이 (TDD 극적 절감)
Scenario B:  SDD input=353 / TDD input=351  → 0.6% 차이 (거의 동일)
```

**Scenario B에서 TDD input이 SDD와 같은 이유:**
DNA 분석기의 테스트는 ORF 탐색, reverse complement 검증 등 복잡한 생물학적 로직을 포함해야 함.
Claude는 구현 코드를 쓰지 않는 대신, `tmp_path` fixture, 합성 서열 생성, 좌표 경계 검증 등
정교한 테스트를 작성하느라 SDD와 동등한 컨텍스트를 소비함.

**TDD output이 SDD보다 많은 역전 현상 (B 시나리오):**
- SDD-B output: ~2.9k / TDD-B output: ~3.3k
- 테스트 코드(214줄)가 구현 코드(92줄)보다 2.3배 길어짐
- "스텁을 짧게, 테스트를 충실히" 전략이 테스트 복잡도에 따라 output 절감 효과를 상쇄시킴

**결론: TDD 토큰 절감은 테스트 작성 난이도에 반비례한다.**

## 6. 종합 결론

### 토큰 관점
> **TDD가 유리하지만, 효과는 과제 성격에 따라 크게 다르다.**

- **단순/구조적 과제(A)**: 테스트가 쉽고 구현이 복잡 → TDD 40% 절감. 명확한 승자.
- **복잡/도메인 과제(B)**: 테스트 자체가 복잡 → TDD 9.5% 절감에 그침. 테스트 output이 구현 절감을 상쇄.

### 품질 관점
> **도메인 친숙도가 pi 코드 품질을 결정한다.**

- CSV 통계(범용): Claude(SDD) 압도적 우위 (Halstead 2.2×)
- DNA 분석(도메인): pi(TDD) 오히려 우위 (Halstead 1.6×, MI 높음)
- pi가 생물학 관례(FASTA ID 파싱)를 Claude보다 잘 적용한 것은 Qwen3.5의 훈련 데이터 특성으로 추정

### 실용 권고

| 상황 | 추천 |
|------|------|
| 비용이 최우선, 테스트 로직 단순 | **TDD** |
| 코드 품질 최우선, 범용 도메인 | **SDD** |
| 도메인 특화 코드 (바이오, 수학 등) | **TDD** (pi가 도메인 지식 활용) |
| 테스트 자체가 복잡한 과제 | **SDD** (TDD 이점 희석) |

### 한계 및 향후 과제
- 각 시나리오 1회씩만 실험 — 노이즈 가능성 있음
- output 토큰이 `/cost`에서 `~k` 단위로만 표시되어 정확한 비교 불가
- pi(Qwen3.5) 모델을 더 강력한 모델로 교체 시 품질 격차가 달라질 수 있음
- 시나리오 추가(비동기, DB, ML 파이프라인)로 패턴 일반화 필요
